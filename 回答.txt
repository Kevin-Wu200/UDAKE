UDAKE 企业级安全策略评审与采纳决策（最终确认版）

一、产品密钥校验位算法

【决策】
部分采纳（优化后使用）

【不采纳方案】
- Luhn 算法（仅适用于纯数字）
- 简单求和（安全性不足）
- CRC 校验（不适用于人工输入场景）

【最终方案】
- 校验位位置：密钥最后一位
- 算法：Base36 + 加权校验算法（自定义实现）

【理由】
- 支持数字与大写字母混合
- 能有效检测输入错误
- 实现复杂度适中，适合工程落地


二、Redis 使用策略

【决策】
全面采纳（结构优化）

【采纳内容】
- Token 黑名单机制
- 邮箱验证码存储（10分钟过期）
- 产品密钥缓存（减少数据库查询）

【不采纳内容】
- 不使用 Redis 存储 Session（系统采用 JWT 无状态认证）

【持久化策略】
- 使用 AOF + RDB 混合模式（提高数据安全性）


三、UI 组件库选择

【决策】
采纳 Element Plus

【理由】
- Vue 3 原生支持
- 中文文档完善
- 适合管理后台快速开发


四、企业密钥名额释放机制

【决策】
采纳（增加强一致性约束）

【最终方案】
- 用户删除后立即释放名额

【强制要求】
- 用户删除操作与名额释放必须在同一数据库事务中完成

【风险说明】
- 若不使用事务，会导致名额数据不一致问题


五、多设备登录管理

【决策】
采纳（增强安全设计）

【最终方案】
- 支持多设备同时登录
- 提供设备列表查看功能
- 支持手动踢出设备

【关键补充】
- 每个 Token 必须包含：
  - device_id
  - jti（唯一标识）

【说明】
- 否则无法实现精确设备管理与踢出机制


六、找回密码频率限制

【决策】
强化后采纳

【最终方案】
- 每小时最多 3 次
- 每天最多 10 次

【超限处理】
- 暂停找回功能 1 小时
- 记录审计日志

【说明】
- 原方案安全性不足，已加强防刷机制


七、审计日志性能优化

【决策】
完全采纳

【优化措施】
- 按时间分区（推荐 PostgreSQL 原生分区）
- 建立索引：
  - user_id
  - event_type
  - timestamp
- 查询分页（每页 20 条）
- 导出限制（最多 10,000 条）

【推荐实现】
- 使用 PostgreSQL 分区表：
  PARTITION BY RANGE (timestamp)


八、邮箱验证码安全策略

【决策】
部分采纳（优化字符集）

【不推荐】
- 使用大小写混合（易混淆）

【最终方案】
- 字符集：数字 + 大写字母（去除易混淆字符）

【字符集定义】
ABCDEFGHJKLMNPQRSTUVWXYZ23456789

【发送策略】
- 验证码明文显示在邮件中

【重发机制】
- 60 秒冷却时间
- 每次重发生成新验证码


九、数据库连接池配置

【决策】
补充完善后采纳

【推荐配置】
- pool_size = 10
- max_overflow = 20
- pool_timeout = 30
- pool_recycle = 1800

【必须配置】
- 启用连接健康检查（pre_ping = True）

【说明】
- 提升系统稳定性与数据库连接可靠性


十、前端路由与状态管理

- 不共享登录状态
