# 安全加固影响范围评估

> 创建日期：2026-05-23
> 状态：已完成评估

## 一、数据库字段加密影响评估

### 1.1 受影响模块

| 模块 | 影响级别 | 说明 |
|------|---------|------|
| `auth_db/models.py` (ProductKey) | 高 | product_key 字段新增 ciphertext 和 hash 索引字段 |
| `auth_db/models.py` (User) | 低 | email 字段评估为暂不加密（需 Email 服务适配） |
| `api/product_keys_api.py` | 中 | 密钥读写需同时处理明文/密文两种格式 |
| `services/数据安全服务.py` | 低 | 已有 encrypt_field/decrypt_field，无需改动 |
| `auth/security.py` | 低 | AES-GCM 原语已就绪 |

### 1.2 性能影响评估

| 操作 | 预期延迟增加 | 备注 |
|------|-------------|------|
| 单次加密 | <1ms | AES-GCM 硬件加速 |
| 单次解密 | <1ms | 同上 |
| 批量读取（100条） | <5ms | 批量解密可并行 |
| 密钥查找（by hash） | <1ms | SHA-256 哈希索引 |

**结论**：加解密延迟控制在 5ms 以内，满足性能要求。

### 1.3 密钥版本管理策略

- 每个加密字段前缀 `kmsf:v1:{key_id}:...` 标识加密密钥版本
- 密钥轮换时旧数据仍可按 key_id 索引到对应密钥解密
- 建议密钥轮换周期：6 个月
- 旧密钥保留 2 个周期（12 个月）后归档

### 1.4 回滚策略

1. **迁移前**：全量备份 product_keys 表
2. **迁移中**：使用 `--dry-run` 先试运行验证
3. **迁移后**：运行 `--verify` 确认数据一致性
4. **紧急回滚**：执行 `--rollback` 将密文还原为明文
5. **回滚时限**：迁移后 72 小时内可无损回滚

## 二、配置热更新影响评估

### 2.1 受影响模块

| 模块 | 影响级别 | 说明 |
|------|---------|------|
| `runtime_config_manager.py` | 高 | 新增 watchdog/sighup 支持 |
| `main.py` | 中 | lifespan 中初始化热重载 |
| `api/product_keys_api.py` | 中 | 运行时配置随重载生效 |
| `api/配置接口.py` | 低 | API 层配置管理已对接 RuntimeConfigManager |
| `services/cache_service.py` | 低 | 缓存 TTL 需注册重载回调 |
| `security/ip_reputation.py` | 低 | IP 信誉阈值需注册重载回调 |

### 2.2 并发安全分析

- `RuntimeConfigManager` 使用 `threading.RLock` 保护版本列表
- `check_reload()` 通过 mtime 检测文件变更，原子替换版本
- 回调函数在锁外执行，避免死锁
- 已注册的回调列表在锁内拷贝后执行，回调异常不影响主流程

### 2.3 回滚策略

1. **API 回滚**：通过 `/api/config/rollback` 接口按版本号回滚
2. **手动回滚**：直接编辑 YAML 文件，watchdog 自动检测并重载
3. **SIGHUP 重载**：`kill -HUP <pid>` 手动触发
4. **紧急恢复**：替换为最后已知正常的配置文件备份

## 三、审计日志完整性影响评估

### 3.1 新增功能

- `sign_audit_logs()`：生成 HMAC-SHA256 签名链
- `verify_audit_integrity()`：校验签名链完整性
- 签名链最多覆盖最近 1000 条日志
- 根签名可存储到外部系统作为信任锚点

### 3.2 性能影响

- 签名计算：每条日志 ~0.1ms（HMAC-SHA256）
- 校验计算：与签名相同
- 内存开销：每个签名 64 字符 hex 字符串

## 四、ABAC 策略增强影响评估

### 4.1 新增策略

| 策略 ID | 操作 | 效果 | 条件 |
|---------|------|------|------|
| product_key_create_policy | product_key:create | deny | user.clearance=low, resource.classification=enterprise |
| product_key_revoke_policy | product_key:revoke | allow | user.role=security_admin |
| audit_log_full_access_policy | audit:read_full | allow | user.role=auditor |
| user_data_privacy_policy | data:read_sensitive | deny | user.role=guest |
| security_admin_full_access_policy | * | allow | user.role=security_admin |

### 4.2 兼容性

- 新增策略不影响现有 RBAC 权限
- ABAC 和 RBAC 为 AND 关系（两者都允许才放行）
- 无 ABAC 策略匹配默认放行（向后兼容）

## 五、总体风险评估

| 风险项 | 概率 | 影响 | 缓解措施 |
|--------|------|------|---------|
| 加密迁移数据丢失 | 低 | 高 | 先备份、dry-run、验证 |
| 密钥丢失导致数据不可恢复 | 低 | 严重 | KMS 密钥轮换保留旧密钥 |
| 热重载并发竞态 | 低 | 中 | RLock + 原子版本切换 |
| 审计日志签名开销 | 低 | 低 | 仅签名最近 1000 条 |
| ABAC 策略误拦截 | 低 | 中 | 默认放行，策略显式 deny |

**总体结论**：安全加固改造风险可控，建议分阶段实施，先执行加密迁移试运行和配置热重载灰度测试。
