# 前端国际化开发与维护指南

## 1. 目标
- 统一前端国际化入口为 `apps/frontend/js/utils/I18n.ts`。
- 语言资源按 `模块.功能.键名` 命名。
- 所有新增 UI 文案必须通过 `I18n` 输出。

## 2. 语言资源组织
- 基础语言：`I18n.ts` 内置 `zh-CN`、`en-US`。
- 扩展语言：`apps/frontend/js/locales/*.json`（当前包含 `zh-TW`、`ja-JP`、`ko-KR`）。
- 懒加载：切换到扩展语言时由 `I18n.setLocaleAsync` 自动加载并缓存。

## 3. 开发规范
- 页面静态文本使用 `data-i18n` / `data-i18n-title` / `data-i18n-placeholder`。
- 代码中使用：
  - `I18n.t(key, params)` 普通翻译。
  - `I18n.tp(baseKey, count, params)` 复数翻译。
  - `I18n.tv(baseKey, variant, params)` 变体翻译。
- 格式化能力：
  - `I18n.formatDate / formatDateTime`
  - `I18n.formatNumber / formatCurrency`
  - `I18n.formatRelativeTime`
  - `I18n.sortByLocale`

## 4. 翻译工作流（团队协作）
- 导出基准键：以 `zh-CN` 为基准维护键集合。
- 翻译分支流程：
  1. 业务分支新增键到 `zh-CN` 与 `en-US`。
  2. 运行 `npm run i18n:check` 生成差异报告。
  3. 翻译同学基于 `reports/i18n/i18n-report.json` 补齐扩展语言。
  4. 评审要求至少 1 名业务评审 + 1 名前端评审。
- 版本管理：翻译键随代码走 Git PR，不使用脱离代码库的手工版本。

## 5. 自动化检查
- 本地检查：`npm run i18n:check`
- 严格检查：`npm run i18n:check:strict`
- CI 工作流：`.github/workflows/i18n_check.yml`
  - 检查缺失翻译键
  - 检查命名规范
  - 检查可疑硬编码文本
  - 输出覆盖率与未使用键报告
- 审查基线：`configs/i18n-baseline.json`
  - `reservedUnusedKeys`：已评审允许保留的未使用键。
  - `reviewedHardcoded`：已评审通过的历史硬编码记录（按文件+行+文本精确匹配）。

## 6. 翻译管理系统集成
- 统一翻译管理入口：`translation-management/`。
- 导出翻译源：`npm run i18n:tms:export`
  - 输出 `translation-management/source/zh-CN.json` 和 `translation-management/source/en-US.json`。
- 校验翻译完整性：`npm run i18n:tms:validate`
  - 报告文件：`translation-management/reports/validation.json`。
- 严格校验（CI 推荐）：`npm run i18n:tms:validate:strict`
- 外部平台对接：
  - 已提供 Crowdin 配置：`.crowdin.yml`。
  - 设置 `CROWDIN_PROJECT_ID` 与 `CROWDIN_PERSONAL_TOKEN` 后即可接入平台同步流程。

## 7. 持续维护
- 每周定时 CI 产出报告（工作流 `schedule`）。
- 每次需求迭代完成后清理未使用键。
- 保持语言切换组件和设置面板一致支持全部已发布语言。
- 对关键流程（上传、插值、导出、错误提示）做双语回归测试。
