## 一、概述

### 1.1 背景

UDAKE 平台已部署 Prometheus + Alertmanager + Grafana 基础监控体系，但存在以下痛点：
- 告警规则依赖人工阈值设定，误报率高
- 日志异常靠人工排查，响应滞后
- 故障恢复依赖手动操作，无自动化自愈能力

### 1.2 目标

引入 AIOps 能力，实现：
1. 智能日志异常检测（准确率 ≥ 90%）
2. 告警智能降噪与聚合（降噪率 ≥ 70%）
3. 自动化故障自愈 Pipeline

│       │             │              │          │
│  ┌────┴─────────────┴──────────────┴───────┐  │
│  │        告警聚合与降噪引擎                │  │
│  └────────────────┬───────────────────────┘  │

### 3.2 组件说明

| 组件 | 功能 | 技术选型 |
|------|------|---------|
| 日志异常检测 | 自动识别日志中的异常模式 | 基于 LLM 的日志解析 + Isolation Forest |
| 指标异常检测 | 检测 CPU/内存/延迟等指标异常波动 | Prophet 时序预测 + 3-sigma 动态阈值 |
| 根因分析 | 故障发生时自动定位根因 | 因果推断 + 服务依赖拓扑 |
| 告警降噪 | 减少重复告警，聚合关联告警 | 时间窗口聚合 + 语义相似度去重 |
| 故障自愈 | 自动执行预定义恢复动作 | Argo Workflows + 自定义 Playbook |

## 四、日志异常检测方案

### 4.1 方案选型

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| 规则匹配 | 精确、快速 | 无法发现未知异常 | 已知错误模式 |
| 统计模型 (3-sigma) | 简单、可解释 | 对周期性数据效果差 | 稳定日志量 |
| ML 模型 (Isolation Forest) | 无需标注数据 | 高维稀疏数据效果差 | 通用异常检测 |
| LLM 日志解析 | 理解上下文语义 | 成本高、延迟大 | 复杂异常研判 |

### 4.2 推荐方案：混合模型

```
日志流 → 结构化解析 → 特征提取 → 多模型投票
                                    ├── Isolation Forest (统计异常)
                                    ├── 日志模板聚类 (模式异常)
                                    └── LLM 二级研判 (关键告警)
```

### 4.3 检测流程

1. **日志采集**：Filebeat → Kafka/Loki
2. **结构化解析**：Drain3 算法提取日志模板
3. **特征工程**：日志频率、错误率、新模板出现率
4. **异常检测**：Isolation Forest 检测统计异常
5. **LLM 研判**（可选）：异常 event 触发 LLM 分析上下文

## 五、指标异常检测方案

### 5.1 检测策略

| 指标类型 | 检测方法 | 训练周期 |
|---------|---------|---------|
| 请求延迟 (P50/P95/P99) | Prophet 预测 + 动态阈值 | 每日 |
| 错误率 | 3-sigma 滚动窗口 | 实时 |
| CPU/内存使用率 | 趋势检测 + 突变检测 | 每周 |
| 任务队列深度 | 季节性分解 + 残差异常 | 每日 |
| 自定义业务指标 | 孤立森林 | 每日 |

### 5.2 Prophet 时序预测

```python
# 伪代码示例
from prophet import Prophet
model = Prophet(interval_width=0.99)  # 99% 置信区间
model.fit(historical_metrics)
forecast = model.predict(future_periods)

# 超出置信区间的点标记为异常
anomalies = actual[actual > forecast['yhat_upper']]
```

## 六、告警降噪方案

### 6.1 降噪策略

```
原始告警 (100条/分钟)
    │
    ├── 时间窗口聚合 (相同告警 5 分钟内合并)
    │   └── 降噪率: ~30%
    │
    ├── 拓扑关联聚合 (同一服务的上下游告警关联)
    │   └── 降噪率: ~25%
    │
    ├── 语义相似度去重 (基于告警描述文本相似度)
    │   └── 降噪率: ~15%
    │
    └── 静默规则 (维护窗口、已知问题)
        └── 降噪率: ~10%
                    ↓
        最终告警 (30条/分钟) → 降噪率 ~70%
```

### 6.2 实现方式

- Alertmanager 分组 + 抑制规则（基础降噪）
- 自定义告警聚合服务（高级降噪）
- 告警收敛时间窗口：5 分钟

## 七、故障自愈 Pipeline

### 7.1 自愈分级

| 级别 | 故障类型 | 自愈动作 | 人工介入 |
|------|---------|---------|---------|
| L1 | Pod 重启 | 自动重启（K8s 原生） | 不需要 |
| L2 | 服务降级 | 自动扩容 + 限流开启 | 通知即可 |
| L3 | 数据库连接池耗尽 | 自动重启服务 + 连接池调优 | 需要确认 |
| L4 | 磁盘/内存不足 | 自动清理临时文件 + 扩容 | 需要确认 |
| L5 | 数据损坏 | 自动切换备份 + 告警 | 必须介入 |

### 7.2 自愈工作流

```yaml
# Argo Workflow 示例
apiVersion: argoproj.io/v1alpha1
kind: WorkflowTemplate
metadata:
  name: udake-auto-heal
spec:
  entrypoint: heal
  templates:
    - name: heal
      steps:
        - - name: diagnose
            template: diagnosis
        - - name: restart-service
            template: rolling-restart
            when: "{{steps.diagnose.outputs.result}} == service_degraded"
        - - name: scale-up
            template: hpa-scale
            when: "{{steps.diagnose.outputs.result}} == high_load"
        - - name: notify
            template: send-notification
```

## 八、与现有监控体系的集成

### 8.1 集成方式

| 现有组件 | AIOps 集成方式 |
|---------|---------------|
| Prometheus | 通过 Prometheus API 拉取历史指标训练模型 |
| Alertmanager | AIOps 引擎作为 Alertmanager webhook receiver |
| Grafana | 新增 AIOps 面板（异常检测结果、自愈历史） |
| Loki | 日志异常检测模型的输入源 |
| 企业微信/钉钉 | 降噪后的告警 + 自愈结果推送 |

### 8.2 渐进式上线方案

1. **Phase 1**：指标异常检测 + 告警降噪（与现有流程并行运行 2 周）
2. **Phase 2**：日志异常检测 + L1/L2 自动自愈
3. **Phase 3**：根因分析 + L3/L4 半自动自愈

| 异常检测准确率 | ≥ 90% | 人工标注验证集 |
